## What This Pattern Is

Independent minds — or independent *sessions* — produce the same idea
near-simultaneously, with no contact. A shared context constrains the
possibility-space so the same step becomes *available and probable* (the
"adjacent possible"); independent selection then makes it likely several
reach it at once. No transmission, no field: structural availability +
independent selection.

Sharply distinct from the mystical versions (morphic resonance, noosphere,
collective unconscious), which posit *transmission*; this posits
*constrained availability*.

## Recognizing When This Applies

**Conditions:**
- Two parallel sessions (or collaborators) reach the same design with no
  contact — read it as evidence the step was contextually forced, which is
  *validation*, not coincidence or leakage.
- Priority disputes, "great minds" moments, simultaneous-invention stories.
- Calibrating an originality claim: if the context makes an idea adjacent,
  expect multiples — lower the surprise and the claim.

**Not this pattern:**
- Convergence with actual contact (shared draft, shared chat, common
  upstream source) — that's transmission, the boring explanation; rule it
  out first.
- Convergence on *vocabulary* without convergence on mechanism.

## Cross-Domain Verification

**Mechanism check:** shared causal structure — context constrains the
option-space; selection over the same constrained space yields the same
pick independently. Not a vocabulary resemblance.

**Domains verified:**
- Science — Newton/Leibniz calculus; Darwin/Wallace selection.
- Invention — the telephone (Bell/Gray, same day at the patent office).
- Mathematics — repeated independent proofs ("multiples" are the norm,
  per Merton, not the exception).
- AI practice — parallel Claude sessions on this project independently
  reaching the same design conclusions (documented occurrence, 2026-06).

**Lineage:** Merton ("multiples" in scientific discovery); Kauffman /
Steven Johnson (the adjacent possible).

## What Makes It Work

1. **Transmission ruled out first** — the pattern only bites once contact
   is excluded; otherwise it's an excuse for not checking.
2. **The context does the explaining** — name *which* shared constraints
   made the step adjacent; if you can't, the convergence is unexplained,
   not explained-by-this-pattern.
3. **Use it both directions** — as humility (your idea was adjacent, expect
   co-discoverers) and as validation (two blind parties converging means
   the constraints, not a bias, picked the answer).

---

## Testing This Pattern

Before relying on this:
1. **Baseline:** Encounter a convergence WITHOUT consciously invoking the pattern.
2. **With pattern:** On another, ask "what shared constraints made this step adjacent?"
3. **Compare:** Did the question produce a checkable account vs a shrug?
4. **Pressure:** Does it work when rushed?

**Honest note:** the library's A/B protocol hasn't been run (n=0
in-session); the mechanism rests on the named lineage and the documented
occurrences, including one live in this project's own multi-session work.
