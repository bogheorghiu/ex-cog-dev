## What This Pattern Is

Applying a method or principle **to itself** is both a *test* (does it
survive its own standard?) and a *generator* (it exposes reflexive
structure, or hidden inconsistency). A claim that ranges over a domain
*includes itself* if it's in that domain; consistency then *requires* it
survive self-application — and the act reveals either robustness or
paradox.

**Origin:** Staged publicly as ex-cog-dev issue #94 (Jun 7, 2026; snapshot
in the excogitations archive), status "reconcile with local library" —
reconciled 2026-06-12. The library had members of this family but not the
general form (see kin below).

## Recognizing When This Applies

**Conditions:**
- Shipping a universal rule, style guide, eval framework, or debiasing
  method — run it on itself first.
- A methodology document that doesn't follow its own methodology — the
  inconsistency is the finding.
- Designing a test/audit: the auditor is in the audited domain (a security
  tool is itself a system with a POSIWID; a debiasing pass can itself
  introduce bias).

**Not this pattern:**
- Methods that genuinely don't range over themselves (a recipe isn't food;
  no inconsistency in a terse style guide about verbose prose *unless* it
  claims universality over all writing including itself).
- Reflexivity for its own sake — the move earns its keep when it changes
  the artifact, not as cleverness.

## Cross-Domain Verification

**Mechanism check:** genuine logical mechanism — the self-reference family
(the liar, Gödel, reflective equilibrium). Self-inclusion is entailed by
the claim's own scope; nothing metaphorical transfers.

**Domains verified (several from this project's own record):**
- `skill-activation-testing` tested by its own method.
- The description-writing guidance audited on its own description.
- POSIWID applied to a security tool meant to protect (what does the
  protector *do*?).
- cui-bono's recursive debiasing step — "did my debiasing itself introduce
  a new bias?"
- The `rule-design` rule built *by* the compose-don't-fork principle it
  preaches.
- Classical: reflective equilibrium in ethics; Gödel-style limits in logic.

**Kin in this library** (instances of the family, narrower than the
general form): `description-becomes-embodiment` (a description that
performs the practice it describes — reflexivity as pedagogy);
`pattern-recognition-witness` (watching your own debugging — reflexivity
turned on one's process). Held as kin, not merged: each has its own
mechanism detail and antidote.

## What Makes It Work

1. **Scope honesty** — first establish the claim actually includes itself;
   forced reflexivity on a non-self-including claim is a category error.
2. **Both outcomes are information** — surviving self-application is
   evidence of robustness; failing it is a generator (the failure points at
   the hidden premise or the needed exception).
3. **Run it early** — self-application at design time is cheap; a shipped
   universal rule that fails itself is an authority leak others will find.

---

## Testing This Pattern

Before relying on this:
1. **Baseline:** Ship a rule/framework WITHOUT the self-pass.
2. **With pattern:** On the next, run it on itself before shipping.
3. **Compare:** Did the self-pass surface changes the plain review missed?
4. **Pressure:** Does it work when rushed?

**Honest note:** the library A/B protocol hasn't been run as such, but
unlike most candidates this one has repeated *documented in-project use*
(the occurrences above) — the n=0 caveat applies to the controlled
comparison, not to field use.
